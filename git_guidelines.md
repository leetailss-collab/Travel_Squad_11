# 🚀 Travel Squad 11 - Git 동기화 및 개발 주의사항 가이드 (실수 방지)

이 문서는 여러 기기(예: 집 컴퓨터, 회사 컴퓨터)에서 협업하거나 Git Pull/Push를 진행할 때 발생할 수 있는 **데이터베이스 유실, 동기화 오류, 이미지 엑박 현상**을 방지하기 위한 필수 가이드라인입니다.

개발 작업을 시작하거나 코드를 올리고 내려받을 때 반드시 아래 수칙을 확인해 주세요.

---

## 📌 1. 핵심 요약 테이블

| 항목 | Git 추적 여부 | 위험 요소 | 예방 및 해결책 |
| :--- | :---: | :--- | :--- |
| **`plans.json`**<br>(로컬 DB 파일) | **O** (추적 중) | `git pull` 시 로컬 데이터 덮어쓰기 혹은 JSON 코드 충돌 발생 | 1. Pull 받기 전 로컬 파일 백업<br>2. 가급적 Firestore 동기화 완료 후 커밋 |
| **`uploads/`**<br>(업로드 이미지) | **X** (제외됨) | 기기 이동 시 업로드한 이미지가 보이지 않음 (엑박 현상) | 1. 기기 변경 시 `backend/public/uploads` 수동 복사<br>2. (장기 해결책) Cloud Storage 도입 |

## 🔍 2. 세부 주의사항 및 행동 강령

### ⚠️ [주의 1] `plans.json` 변경 및 Git 동기화
* **상황**: `backend/data/plans.json`은 Git 리포지토리에서 관리하는 파일입니다. 로컬 환경에서 DB 데이터가 수정되면 이 파일도 변경될 수 있습니다.
* **위험**: 
  * 로컬에서 변경된 최신 일정을 깃허브에 커밋/푸시하지 않은 상태에서 다른 기기로 넘어가 `git pull`을 받으면 최신 데이터가 유실되거나 덮어씌워질 수 있습니다.
  * 두 명 이상이 동시에 `plans.json`을 수정하고 올릴 경우, 복잡한 JSON 충돌이 발생해 데이터 구조가 깨질 수 있습니다.
* **행동 강령**:
  * **Pull 받기 전**: 항상 작업 중인 `backend/data/plans.json` 파일을 다른 안전한 곳에 **복사(백업)**해 두세요.
  * **동기화 확인**: 데이터 수정 작업이 완료되면 최신 `plans.json`이 서버와 로컬에 정상 저장되었는지 확인한 후 커밋하세요.

---

### 📷 [주의 2] 이미지 업로드와 엑박(Broken Link) 현상
* **상황**: 이미지 파일들은 `backend/public/uploads/` 폴더에 저장되며, 이 폴더는 `.gitignore` 설정에 따라 **Git에 올라가지 않습니다.**
* **위험**: A 컴퓨터에서 일정을 추가하며 업로드한 이미지는 B 컴퓨터로 `git pull`을 받아도 전송되지 않으므로, B 컴퓨터에서는 이미지가 보이지 않는(엑박) 현상이 발생합니다.
* **행동 강령**:
  * 여러 컴퓨터를 오가며 개발할 때, 이미지가 정상적으로 나타나야 하는 경우 A 컴퓨터의 `backend/public/uploads/` 내 파일들을 B 컴퓨터의 동일 경로로 **수동 복사/전송**해 주어야 합니다.
  * *팁: 개발 모드(포트 3000)에서 이미지가 정상적으로 보이려면 로컬 서버(포트 5000)가 정상 구동 중이어야 프록시를 통해 이미지를 불러올 수 있습니다.*

---

### 🔑 [주의 3] Firebase 서비스 계정 키 관리
* **상황**: Firestore 데이터베이스 연결을 위한 `firebase-service-account.json` 파일은 보안상 Git 추적에서 제외(`.gitignore`)되어 있습니다.
* **위험**: 새 컴퓨터에서 리포지토리를 클론한 후 바로 실행하면, 이 파일이 없어 Firestore와의 연동이 실패합니다. 앱은 오프라인 모드로 실행되며, 이때 잘못 저장하면 데이터가 꼬이거나 로컬 스토리지가 초기화될 수 있습니다.
* **행동 강령**:
  * 새 컴퓨터 세팅 시 기존 기기에서 `firebase-service-account.json`을 가져와 `backend/` 루트 폴더 아래에 반드시 넣어주어야 합니다.

## 🛠️ 3. 로컬 서버 실행 가이드 (포트 설정)

* 프론트엔드와 백엔드가 서로 다른 포트에서 유기적으로 작동하므로, 포트 설정이 어긋나면 업로드나 동기화가 실패합니다.
* **실행 방법**: 반드시 프로젝트 루트의 **`run_local.bat`** 파일을 실행해 주세요.
  * **프론트엔드**: `http://localhost:3000` (Vite dev server)
  * **백엔드**: `http://localhost:5000` (Express)
  * Vite config의 proxy 설정(`/api` -> `http://localhost:5000`)에 의해 모든 API 요청과 업로드가 정상적으로 백엔드로 전달됩니다.

## 🚨 4. 비상 상황 발생 시 복구 요령 (로컬 백업)

만약 깃허브 동기화 과정에서 실수로 데이터베이스를 덮어썼거나 초기화되었더라도, 브라우저가 켜져 있다면 로컬 백업본을 찾을 수 있습니다.

1. 크롬 등 브라우저에서 `F12`를 눌러 **개발자 도구**를 엽니다.
2. 상단 탭에서 **Application(애플리케이션)**을 선택합니다.
3. 좌측 메뉴에서 **Storage -> Local Storage -> http://localhost:3000**을 클릭합니다.
4. `family_travel_plans` 키값에 저장된 JSON 데이터를 찾습니다. (이 데이터가 날아가기 전의 브라우저 로컬 데이터베이스입니다.)
5. 이 데이터를 복사하여 `backend/data/plans.json`에 덮어씌운 후, 백엔드 스크립트를 통해 Firestore로 다시 마이그레이션(동기화)하면 원래대로 복구할 수 있습니다.
